# Proposal: 改进 AI 聊天体验

## Why

当前 AI 聊天功能存在严重的用户体验问题，影响了用户与 AI 助手的交互效率和满意度：

1. **UI 不一致**：TDesign 迁移到 Ant Design 后，部分组件样式未统一
2. **交互问题**：
   - Enter 键无法发送消息
   - 输入内容后发送按钮仍然 disabled
   - Tab 图标和文字间距过大
3. **消息混乱**：工具调用的中间结果被错误保存为用户消息
4. **缺少反馈**：AI 处理时没有明确的加载状态
5. **滚动问题**：新消息不会自动滚动到底部，切换 Tab 也不会滚动
6. **响应冗长**：AI 画完流程图后输出大量冗余说明
7. **布局问题**：BPMN 编辑器高度无法正确填充
8. **缺少过程反馈**：AI 执行工具时，用户看不到详细的操作过程
9. **响应格式单一**：AI 回复为纯文本，不支持 Markdown 格式化
10. **加载状态冗余**：显示两个消息气泡
    - 第一个：独立的 loading 气泡（`isLoading` 控制），显示"AI 正在思考..."
    - 第二个：流式消息内的 loading，显示"AI 正在处理..."
11. **过程信息丢失**：AI 完成后，操作过程被完全替换，用户无法回顾操作细节
12. **Loading 状态不同步**：流式消息完成后，`isStreaming` 未设置为 false，导致 loading 一直显示
13. **视觉分离问题** (问题 10-12 修复后发现)：
    - 过程信息（progress logs）和 loading 状态在视觉上看起来像两个独立的气泡
    - 原因：`.progress-logs` 和 `.streaming-loading` 都有独立的背景色、边框、圆角和内边距
    - 预期：应该在同一个统一的聊天消息气泡内，过程信息和 loading 是气泡内的不同区域
14. **消息持久化不一致** (问题 13 修复后发现)：
    - 运行时：显示处理后的消息内容（如 API 返回为空时显示"✅ 操作已完成"）
    - 刷新后：从数据库加载的是原始 API 响应（可能为空字符串）
    - 原因：数据库保存的是 `claudeService.sendMessage()` 返回的原始响应，而不是 `finalizeMessage()` 显示的处理后内容
    - 结果：运行时和历史消息内容不一致，用户体验差

## What Changes

本次变更将全面优化 AI 聊天界面，包括以下核心改进：

1. **统一 UI 风格**：所有 AI 聊天相关组件采用 Ant Design 设计规范
2. **修复交互**：确保输入、发送、滚动等基础功能正常工作
3. **优化消息流**：只保存用户消息和 AI 最终响应，不保存中间工具调用
4. **改善反馈**：添加明确的加载状态指示
5. **简化响应**：AI 完成操作后简短确认即可
6. **增强过程可见性**：实时显示 AI 执行的每个操作步骤
7. **支持富文本响应**：使用 Markdown 渲染最终响应，支持列表、代码、链接等格式
8. **统一 Loading 状态**：
   - 移除独立的 loading 气泡（`isLoading` 控制的那个）
   - 只在流式消息内部显示 loading
   - 确保只有一个消息气泡
9. **保留操作历史**：AI 完成后保留操作过程，在底部追加 Markdown 摘要
10. **正确的状态管理**：确保 `isStreaming` 在 AI 完成后正确设置为 false
11. **统一消息气泡视觉**：
   - 过程信息和 loading 状态应在同一个聊天消息气泡内
   - 气泡有统一的外层边框、背景和圆角
   - 内部区域（过程日志、loading、Markdown）通过背景色或分隔线区分，但不应有独立的边框
12. **消息持久化一致性**：
   - 数据库保存的应该是最终显示给用户的消息内容
   - 不应保存原始 API 响应，而应保存处理后的内容
   - 确保运行时显示和历史加载的内容完全一致

## 风险与缓解

### 风险
1. **消息历史丢失**：修改消息保存逻辑可能影响现有聊天记录
2. **组件通信复杂**：父子组件间的消息传递可能出错

### 缓解措施
1. 只修改新消息的保存逻辑，不影响历史记录加载
2. 使用 Vue 标准的 `ref` 和 `defineExpose` 机制，避免 DOM 查询
3. 添加详细的日志和错误处理

## 替代方案

### 方案 1：保留完整对话历史
将工具调用也保存到数据库，但添加 `hidden` 标记不显示

**优点**：完整的对话追溯
**缺点**：数据库存储增加，UI 需要复杂的过滤逻辑

**决策**：不采用，用户只关心最终结果

### 方案 2：使用事件总线通信
用 EventBus 替代 ref 进行组件通信

**优点**：解耦更好
**缺点**：调试困难，难以追踪消息流

**决策**：不采用，Vue ref 机制已足够清晰
